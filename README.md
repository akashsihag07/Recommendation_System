# ReelMatch — Movie Recommendation Service

A small web app that recommends movies based on what you're in the mood for. You
pick some genres, set a minimum rating and a release-year range, optionally name
a movie you love, and it returns a ranked list of films with a short reason for
each match.

This was built for the Matching / Recommendation project. Everything runs in
Docker — one command and it's up.

---

## Why I chose this project

Out of the three options (matching, recommendation, image recognition) I went
with a recommendation service because the logic is something I could fully
understand and explain rather than treat as a black box. With image recognition
I'd be leaning on a pre-trained model I didn't build; here, every reason a movie
shows up in your results is something I can point to in the code.

Movies were a natural fit because there's a well-known public dataset (TMDB) with
genres, ratings and descriptions, and because the results are easy to sanity
check — if you ask for highly-rated action/adventure from the last decade and it
hands you Mad Max and Guardians of the Galaxy, you can tell it's working.

## What's special about it

- **Explainable matches.** Each card shows *why* it matched ("Matches 2/3 of your
  genres · 7.9★") instead of just dropping a list on you.
- **A "more like this" mode.** Name a film you love and the recommender blends in
  how similar each candidate is to it (genre + keyword overlap), not just your
  filters.
- **Fully self-contained.** No external image API or API keys. Since the dataset
  has no posters, each card gets a colored gradient generated from its title, so
  the whole thing works offline after the first build.

## Running it

You only need Docker installed.

```bash
git clone <your-repo-url>
cd movie-recommender
docker compose up --build
```

Then open:

- **App:** http://localhost:3000
- **API docs (auto-generated by FastAPI):** http://localhost:8000/docs

On the first start the backend loads ~4,700 movies into MongoDB from the bundled
CSV. That takes a few seconds — if the app shows no genres for a moment, give it
a second and refresh. The data persists in a Docker volume, so later starts are
instant.

To stop:

```bash
docker compose down
```

To wipe the database and reseed from scratch:

```bash
docker compose down -v
```

## How the matching works

It's a **content-based** recommender — no machine learning model, no user
history. For each candidate movie it computes a score from a few signals and
returns the highest-scoring ones. The math lives in `backend/app/scoring.py`.

The final score is a weighted sum (weights in `scoring.py`):

| Signal       | Weight | What it measures                                   |
|--------------|--------|----------------------------------------------------|
| genre match  | 0.50   | how many of your chosen genres the movie has       |
| similarity   | 0.25   | genre + keyword overlap with your favorite movie   |
| rating       | 0.15   | the TMDB rating, scaled to 0–1                     |
| popularity   | 0.10   | log-scaled popularity (so hits don't dominate)     |

Hard filters (minimum rating, year range, and a minimum vote count so obscure
movies with noisy ratings are ignored) are applied as a MongoDB query first, then
the remaining candidates are scored in Python.

The weights were hand-tuned by trying a few values and seeing which results felt
right — they aren't learned from data, which is a fair thing to point out.

## Data

**Source:** [TMDB Movies Dataset](https://www.kaggle.com/datasets/asaniczka/tmdb-movies-dataset-2023-930k-movies)
by asaniczka (Kaggle), sourced from [The Movie Database (TMDB)](https://www.themoviedb.org/).
It's a large, regularly-updated multilingual catalogue. As allowed by the project
rules, the dataset source is declared here.

**Preparation:**

1. From the full dataset I keep movies with at least 25 votes that have both a
   genre and an overview — enough ratings for the score to be meaningful. This
   trims the raw file to a clean working catalogue (exported to
   `backend/data/tmdb_movies_clean.csv`, which ships with the repo so it's
   plug-and-play).
2. In `cleaning.py`, each row is turned into a uniform record: genres and
   keywords (comma-separated text) are split into lists, the year is pulled from
   `release_date`, and the numeric fields are cast.
3. Rows still missing a title, genres, or an overview are dropped, since they
   can't be matched on.

Because it's one dataset, every movie shares the same schema and scale — so there
are no cross-source normalization issues, and `original_language` gives a clean
language filter (English, Hindi, Tamil, and many more).

## API endpoints

| Method | Path              | Purpose                              |
|--------|-------------------|--------------------------------------|
| GET    | `/`               | health check                         |
| GET    | `/genres`         | all genres (used for the form chips) |
| GET    | `/movies/search`  | title autocomplete (`?q=`)           |
| POST   | `/recommend`      | the main one — takes preferences     |

All functions are documented with docstrings, and FastAPI auto-generates
interactive docs at `/docs`.

## Known limitations / what I'd improve

Being honest about the rough edges:

- The score weights are hand-tuned, not learned. A nicer version would A/B test
  them or learn from clicks.
- CORS is wide open (`allow_origins=["*"]`) — fine for a local project, but I'd
  restrict it to the real frontend URL in production.
- MongoDB runs without authentication for local simplicity.
- The recommender scores candidates in Python on each request. It's fast enough
  for ~4,700 movies, but for a much bigger catalogue I'd push more of the work
  into the database or precompute similarities.
